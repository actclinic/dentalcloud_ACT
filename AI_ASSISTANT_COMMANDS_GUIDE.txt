1. MODES OF OPERATION
=====================

ASK MODE (Default):
- Read-only access to information
- Provides dental knowledge and guidance
- Cannot perform CRUD operations
- Example: "What are the symptoms of gingivitis?"

AGENT MODE (Required for actions):
- Full CRUD capabilities
- Can create, update, delete data
- Must be enabled for operational commands
- Toggle using the mode switch in the UI

3. APPOINTMENT OPERATIONS
=========================

CREATE REGISTERED PATIENT APPOINTMENT:
{ "action": "apt_c", "params": { 
  "p_id": "patient123", 
  "dr_id": "doctor456", 
  "dt": "2024-02-15", 
  "t": "10:30", 
  "ty": "Checkup", 
  "status": "Scheduled",
  "branch_name": "Main Clinic",
  "clinical_focus": "Routine dental examination",
  "n": "Patient prefers morning reminders" 
} }

CREATE NEW PATIENT / UNREGISTERED MARKETING LEAD APPOINTMENT:
{ "action": "apt_c", "params": {
  "guest_name": "Aung Aung",
  "guest_phone": "09123456789",
  "guest_source": "Marketing Team",
  "guest_notes": "Caller asked about braces and prefers evening callback",
  "doctor_name": "Dr. Mya",
  "dt": "2026-05-10",
  "t": "14:00",
  "ty": "Consultation",
  "status": "Scheduled",
  "branch_name": "Main Clinic",
  "clinical_focus": "Orthodontic consultation",
  "n": "Lead is not registered yet"
} }

Appointment form mapping: choose Registered Patient for existing patients or New Patient for unregistered leads. Required scheduling basics are date (dt), time (t), and appointment type (ty). Doctor is optional. Branch/location should be included when known, especially in All Branches scope. Status is Scheduled, Completed, or Cancelled and defaults to Scheduled. clinical_focus and n/extra_notes are stored together in the appointment notes as Clinical Focus and Notes. Use guest_name and guest_phone when the person should not become a patient record yet. Do not create a patient first unless the user asks to register or convert the lead. guest_source maps to New Patient Source and guest_notes maps to New Patient Follow-up Notes.

UPDATE APPOINTMENT:
{ "action": "apt_u", "params": { 
  "id": "appointment789", 
  "data": { 
    "time": "11:00", 
    "status": "Confirmed" 
  } 
} }

DELETE APPOINTMENT:
{ "action": "apt_d", "params": { "id": "appointment789" } }

RESCHEDULE APPOINTMENT:
{ "action": "apt_reschedule", "params": { 
  "id": "appointment789", 
  "dt": "2024-02-16", 
  "t": "14:00" 
} }

UPDATE APPOINTMENT STATUS:
{ "action": "apt_status", "params": { 
  "id": "appointment789", 
  "status": "Completed" 
} }

Completing a registered-patient appointment applies the configured per-visit clinical fee at most once. The first completed visit uses the new-patient amount; later completed visits use the returning-patient amount. To waive it only when the user explicitly asks:
{ "action": "apt_status", "params": {
  "id": "appointment789",
  "status": "Completed",
  "skip_clinical_fee": true
} }

Appointments use Clinical Focus and Extra Notes. Do not put target teeth on appointments; record teeth on the treatment/clinical record.

Admin Dashboard > Recalls & Cancels is read-only appointment reporting. Upcoming Recalls = future Scheduled registered-patient appointments from Clinical Focus next appointment. Late / No-show = past Scheduled appointments, including New Patient leads without accounts. Cancelled Appointments = all Cancelled appointments with patient or guest names.

FIND PATIENT APPOINTMENTS:
{ "action": "apt_find_patient", "params": { "name": "John Smith" } }

4. TREATMENT PROCEDURES
=======================

RECORD TREATMENT (auto-calculates doctor commission):
{ "action": "tr_create", "params": { 
  "pid": "patient123", 
  "teeth": [18, 19], 
  "desc": "Composite filling for caries", 
  "cost": 150,
  "meds": [
    {"id": "med789", "qty": 1},
    {"id": "med101", "qty": 2}
  ]
} }

Doctor commission follows the doctor's specialization: General/Specialists use commission_percentage (or treatment custom rates); Ortho/Implant/Surgery use commission_per_visit flat amount.

UNDO TREATMENT RECORD:
{ "action": "tr_undo", "params": { 
  "id": "treatment456", 
  "pid": "patient123", 
  "cost": 150 
} }

6. FINANCIAL OPERATIONS
=======================

PROCESS PAYMENT:
{ "action": "fin_pay", "params": { 
  "pid": "patient123", 
  "amt": 150,
  "method": "Cash"
} }

PAYMENT TYPES (required):
- KPay
- WavePay
- Cash
- MMQR
- Debit Card
- Credit Card
- AYA Pay
- UAB Pay

PAYMENT RECEIPT BEHAVIOR:
- Payments receive a stable receipt number.
- Saved payment receipts use an immutable snapshot for accurate reprints.
- The snapshot can preserve the patient/clinic details, payment type and balances, collector, and captured treatment and medicine lines.
- Receipt header title, currency, and A4/55mm/80mm thermal format are shared across devices for new receipts.
- Old saved receipts keep their original snapshot even when settings or patient data later change.
- The receipt picker can include treatments and standalone medicine sales.
- "Recent" / "NEW" means the treatment or medicine date is today, not merely within the last few days.

GET FINANCIAL REPORT:
{ "action": "fin_report", "params": { "period": "weekly" } }
// Options: "daily", "weekly", "monthly"

DETAILED FINANCIAL ANALYSIS:
{ "action": "financial_analysis", "params": { 
  "start_date": "2024-01-01", 
  "end_date": "2024-01-31" 
} }

7. REPORTING AND ANALYTICS
==========================

DETAILED TREATMENT ANALYSIS (READ-ONLY SCREEN):
- Go to Overview > Treatment Mix (Range) and select More Detail.
- The report follows Overview's inclusive Date From, Date To, and Report Scope controls.
- Performed is the number of saved treatment records. It is not appointment count, tooth count, or unique visit count.
- The report shows distinct patients, multiple-treatment-record patient share, recorded treatment production, average value, discounts, FOC treatments, frequency over time, doctor distribution, most-treated teeth, and a searchable/sortable treatment table.
- Multiple-treatment-record patient share is the percentage of distinct patients with more than one saved treatment record in range. Those records may belong to one visit, so it is not automatically a return, retention, repeat-visit, or loyalty rate.
- In the table, Patients is distinct patients for that treatment, Production is recorded treatment value, Average is production per performance, and Share is the treatment's percentage of all performances in the selected scope.
- Discounts and FOC are separate. Unassigned remains visible in the doctor breakdown but is not counted as a distinct doctor.
- All Branches combines matching branch-local services by normalized treatment name. A single branch preserves treatment-type identity and supports legacy name-based records.
- This screen does not report clinical success rates, outcomes, profit, collections, or seasonal comparisons.
- Loli can explain how to use and interpret the screen, but there is no assistant action that opens it. Assistant Practice Data may contain only a recent subset, so use the screen for authoritative selected-range totals.
- No matching records and report-loading failures are different states. Widen the date range for a genuine empty result; use Try again after a loading error.
- Date changes reload Treatment Analysis while it is open. Changing Report Scope returns to Overview; select More Detail again for the new branch scope.

MANAGER EMAIL COMMUNICATION
===========================

SAVE MANAGER EMAIL:
{ "action": "mgr_email_add", "params": { "email": "boss@example.com", "name": "Dr. Boss", "role": "boss", "primary": true } }

LIST MANAGER EMAILS:
{ "action": "mgr_email_list", "params": {} }

REMOVE MANAGER EMAIL:
{ "action": "mgr_email_remove", "params": { "email": "boss@example.com" } }

SEND EMAIL TO MANAGER:
{ "action": "mgr_email_send", "params": { "to": "boss@example.com", "subject": "Daily Summary", "body": "Today we completed 12 treatments and collected 1,200,000 MMK." } }

NOTES:
- If "to" is omitted, the primary (or only) saved manager email is used.
- Sending requires Agent Mode, confirmation, and Email Delivery enabled in Settings.

10. CONFIRMATION AND ERROR HANDLING
===================================

CONFIRMATION RESPONSES:
When the AI asks for confirmation, respond with:
- "Yes"
- "Confirm"  
- "Proceed"
- "OK"
- "Sure"
- "Y"

CANCELLATION RESPONSES:
To cancel a pending action:
- "No"
- "Cancel"
- "Never mind"
- "Abort"

ERROR HANDLING:
Common error scenarios and solutions:

1. "Agent Mode Required":
   Solution: Switch to Agent Mode using the toggle button

2. "Patient not found":
   Solution: Use p_find to locate the correct patient ID first

3. "Invalid JSON format":
   Solution: Check syntax - ensure proper quotes and brackets
   
4. "Insufficient stock":
   Solution: Check inventory levels before prescribing medicines

5. "Doctor not available":
   Solution: Use staff_availability to check schedules first

---

TIPS FOR EFFECTIVE USAGE:
=========================

1. Always enable Agent Mode for operational commands
2. Use p_find before creating new patients to avoid duplicates
3. Check inventory levels before prescribing medicines
4. Use staff_availability to verify doctor schedules
5. For complex workflows, break them into logical steps
6. The AI maintains context during multi-step processes
7. Speak naturally - the AI understands conversational language
8. You can mix voice and text input seamlessly
9. Previous conversation context is preserved across sessions
10. Use confirmation responses naturally ("Yes, proceed" or "Confirm")
